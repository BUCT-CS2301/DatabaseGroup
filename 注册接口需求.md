# 掌上博物馆 App 普通用户注册接口需求

## 一、背景

掌上博物馆 App 端目前有普通用户注册页面，注册方式分为：

* 手机号注册
* 邮箱注册

注册页面字段包括：

```text
手机号或邮箱
用户名
密码
确认密码
```

其中“确认密码”只在前端校验，不需要传给后端。

目前已有的：

```http
POST /api/v1/users
```

更像是后台管理员创建用户接口，需要 token，并且需要传 `role`。这个接口不适合作为 App 普通用户公开注册接口。

所以希望后台新增一个**专门给掌上博物馆 App 普通用户注册使用的接口**。

---

## 二、建议接口路径

建议新增：

```http
POST /api/v1/auth/register
```

或：

```http
POST /api/v1/users/register
```

更推荐：

```http
POST /api/v1/auth/register
```

因为注册属于认证模块，不属于后台管理员用户管理模块。

---

## 三、认证要求

该接口不需要 token。

请求头：

```http
Content-Type: application/json
```

不需要携带：

```http
Authorization: Bearer xxx
```

---

## 四、接口目标

该注册接口需要满足：

1. 不需要管理员 token；
2. 只能创建 App 普通用户；
3. 前端不传 `role`、`roleIds`、`userType` 等权限字段；
4. 后端内部自动给新用户赋普通用户身份；
5. 注册成功后，该账号必须能立刻通过现有登录接口登录；
6. 注册失败时返回明确错误信息，例如用户名已存在、手机号已存在、邮箱格式错误等。

---

## 五、请求体设计

### 1. 手机号注册

```json
{
  "username": "test1",
  "password": "123456",
  "phone": "17205865351"
}
```


### 2. 邮箱注册

```json
{
  "username": "test2",
  "password": "123456",
  "email": "test2@example.com"
}
```


---

## 六、字段说明

| 字段       | 类型     | 是否必填     | 说明                    |
| -------- | ------ | -------- | --------------------- |
| username | string | 必填       | 用户名，用于展示，也用于后续登录      |
| password | string | 必填       | 密码                    |
| phone    | string | 手机号注册时必填 | 手机号                   |
| email    | string | 邮箱注册时必填  | 邮箱                    |
| nickname | string | 选填       | 昵称，不传时建议默认等于 username |

要求：

```text
phone 和 email 至少填写一个。
```

如果用户选择手机号注册，前端只传 `phone`。
如果用户选择邮箱注册，前端只传 `email`。

---

## 七、前端不希望传的字段

App 普通用户注册接口中，前端不希望传以下字段：

```json
{
  "role": "USER",
  "roleIds": [],
  "userType": "USER"
}
```

原因：

1. App 普通用户注册不应该由前端决定角色；
2. 前端传 `role` 存在权限安全风险；
3. 当前后台的 `user_type` / `role` 相关字段还在调整，前端不适合依赖这些内部实现；
4. 后端应在普通注册接口内部固定赋予“普通用户”身份。

建议后端内部处理为：

```text
新注册 App 用户 -> 自动设置为普通用户角色
```

普通用户在数据库中具体叫：

```text
USER / APP_USER / NORMAL_USER / 普通用户
```

由后台决定即可，App 前端不关心，也不传。

---

## 八、注册成功响应

### 推荐响应

```json
{
  "code": 200,
  "message": "注册成功",
  "data": {
    "objectId": "u-001",
    "username": "test1",
    "nickname": "测试用户1",
    "phone": "17205865351",
    "email": "",
    "avatar": "",
    "createTime": "2026-06-02T10:00:00"
  }
}
```

### 必须返回字段

| 字段       | 说明      |
| -------- | ------- |
| objectId | 用户唯一 ID |
| username | 用户名     |
| nickname | 昵称，可为空  |
| phone    | 手机号，可为空 |
| email    | 邮箱，可为空  |
| avatar   | 头像，可为空  |

`userType` 可以返回，也可以不返回。
如果返回，建议只返回普通用户标识，不要返回复杂权限信息。

---

## 九、注册成功后是否返回 token

注册接口可以不返回 token。

如果注册接口不返回 token，App 前端会在注册成功后自动调用现有登录接口：

```http
POST /api/v1/auth/login
```

使用注册时填写的：

```json
{
  "username": "test1",
  "password": "123456"
}
```

所以需要保证：

```text
注册成功后，立刻可以用 username + password 登录成功。
```

如果后台愿意在注册成功时直接返回 token，也可以，但不是必须。

## 十、App 端注册流程

### 手机号注册流程

```text
用户选择手机号注册
-> 输入手机号、用户名、密码、确认密码
-> 前端校验两次密码是否一致
-> 调用普通用户注册接口
-> 注册成功
-> 前端自动调用现有登录接口
-> 登录成功后进入个人中心
```

### 邮箱注册流程

```text
用户选择邮箱注册
-> 输入邮箱、用户名、密码、确认密码
-> 前端校验两次密码是否一致
-> 调用普通用户注册接口
-> 注册成功
-> 前端自动调用现有登录接口
-> 登录成功后进入个人中心
```
